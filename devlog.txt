APPLICATION OF PYTHON STYLE
Most applicable:
* Imports:
  - "Imports should be grouped in the following order:
    Standard library imports.
    Related third party imports.
    Local application/library specific imports."
  - "You should put a blank line between each group of imports."
* Function and Variable Names:
  - "Function names should be lowercase, with words separated by underscores as necessary to improve readability." (reference __init__.py, line 8. "def login():")
  - "Variable names follow the same convention as function names."
  - "mixedCase is allowed only in contexts where that’s already the prevailing style (e.g. threading.py), to retain backwards compatibility."
* Tabs or Spaces?:
  - "Spaces are the preferred indentation method. Tabs should be used solely to remain consistent with code that is already indented with tabs."
  - "Python disallows mixing tabs and spaces for indentation."
* Limit Line Length:
  - "Limit all lines to a maximum of 79 characters."
Most useful/appreciated by our Devos:
* Indentation:
  - "Use 4 spaces per indentation level."

JAVASCRIPT STYLE
* Which aspects of the style guide were most applicable to your codebase?
  - You may comment out parameters with `array.forEach((value /* , index, array */)` !! This is useful for things like the testing phase where not every parameter may be necessary
  - While if, for and while statements don't require curly braces, you should usually use them .. (get out of bad habits!)
  - Implicit returns with arrow functions! Make our code cleaner!
* Which aspects of the style guide were most useful/appreciated by your Devos?
  - Comment conventions!! We were not aware of the existence of such and will try to make more meaningful comments :)
  - Console.error()??? Prints an error and makes them easier to catch while debugging (woah)
  - i love camelCase
  - The overusage of `var`s pollute the global scope (be careful!) so stick to `let` for variables subject to change and `const` otherwise
  - You shouldn't be using implicit type coercions like "" + int to turn the int into a string :(
    - Instead you should be using String() and Number()!!!!!!!!!!!
